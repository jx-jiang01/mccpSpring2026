## Introduction
The goal of Text-to-SQL is to translate natural language (NL) queries into SQL queries. This technique is especially valuable in business intelligence, where it addresses a core challenge: making data exploration accessible to non-technical users. By enabling anyone to query databases using plain language, Text-to-SQL empowers non-specialists to gain insights from data, much like professional database engineers. At the same time, it enhances the productivity of experts such as data analysts. According to the 2023 Stack Overflow survey, SQL remains one of the most widely used programming languages: over half (51.52%) of professional developers use it at work, yet only about a third (35.29%) have received formal training in it [1].

Recently, Large Language Models (LLMs) have brought transformative advancements to both the research and application of Text-to-SQL techniques. Typically, the user submits natural language questions or requests to a Text-to-SQL system, which then loads the relevant database schema and passes both the schema and the question to the LLM. The system then obtains the generated SQL query from the LLM, and returns this query to the user. In some cases, the system further executes the SQL query on the database and provides the user with the resulting data analysis report.

The emergence of LLM-based Text-to-SQL approaches has led to significant performance improvements. Thanks to advances in LLMs, Text-to-SQL execution accuracy on popular benchmarks, such as the BIRD test set [2], has surged from 7.06 up to 81.67 in just two to three years.

However, relying solely on LLMs for Text-to-SQL remains both costly and inefficient. For example, while providing the entire database schema to the LLM can enhance performance, it also leads to a dramatic increase in token consumption: a few casual queries may involve over 100,000 tokens, causing prohibitive costs for users and organizations. Moreover, current LLMs cannot directly leverage external knowledge (such as knowledge graphs) or real-time information.

To address the aforementioned limitations, Retrieval-Augmented Generation (RAG) has been introduced into Text-to-SQL systems. RAG enhances the quality and relevance of generated SQL queries by incorporating a retrieval mechanism into the generation process. At its core, RAG relies on a retrieval model to search for and integrate useful information with the prompt.

In the context of Text-to-SQL, RAG is typically applied in three main ways: First, it selectively identifies relevant portions of the database schema rather than passing the entire schema to LLMs: greatly reducing token consumption. Second, RAG retrieves supplementary information relevant to the user’s query, improving the overall performance of the system. Third, it provides relevant examples (including example questions, corresponding SQL queries, and any related context) to guide the LLM in generating accurate SQL. This multi-faceted approach not only enriches the contextual information available to the model but also enhances factual accuracy and specificity in the generated queries.

In this paper, we systematically review recent research on RAG-based Text-to-SQL methods from a new perspective. We categorize existing work into two phases: Schema Linking and SQL Generation, according to where RAG is applied. Additionally, we examine research focused on in-context learning and fine-tuning in the Text-to-SQL domain, as these strategies offer valuable insights for optimizing RAG-based approaches.

## Literature Review
Research on LLM-based Text-to-SQL can be broadly categorized into two approaches: in-context learning and fine-tuning. In-context learning focuses on optimizing the prompt to guide the model in performing the task, whereas fine-tuning involves training the LLM on task-specific data to enhance its capabilities. These two approaches are not mutually exclusive; in fact, they are often used together. We distinguish them here to highlight their different focal points in system development.

In-context Learning. Research on in-context learning for Text-to-SQL can be classified into three main prompt strategies: vanilla prompting, decomposition prompting, and chain-of-thought (CoT) prompting. Vanilla prompting involves using zero-shot and few-shot approaches. Zero-shot learning typically explores the effect of different prompt formats and establishes baseline performance for various LLMs. Few-shot learning, on the other hand, incorporates a limited number of examples into the prompt to guide the model: making RAG methods for SQL generation a natural fit within this category. Decomposition prompting addresses complex tasks by breaking them down into more manageable sub-tasks or sub-questions, thereby simplifying the overall problem. For example, DIN-SQL [3] decomposes the process of translating natural language to SQL into multiple steps, ultimately achieving competitive results. Chain-of-thought (CoT) prompting guides LLMs to generate intermediate reasoning steps before producing the final SQL query, resulting in outputs that better align with human expectations. Works like CHASE-SQL [5] introduce strategies such as divide-and-conquer CoT, query plan CoT, and online synthetic example generation, generating multiple SQL queries with a pairwise selection process for the final output. Likewise, CHESS [6] and OpenSearch-SQL [7] leverage CoT prompting to transform natural language questions into accurate SQL queries and achieve strong performance.

Although the in-context learning paradigm achieves impressive accuracy, its computational efficiency remains a concern. Methods that employ multi-stage frameworks or expand the context to boost performance often require more API calls, resulting in significantly higher costs [3]. To address this, in-context learning techniques are frequently combined with RAG approaches in Text-to-SQL systems. For instance, chain-of-thought prompting is widely adopted in RAG-based Text-to-SQL solutions, such as OpenSearch-SQL [7], effectively balancing accuracy and efficiency.

Fine-tuning. Fine-tuning (FT) is a widely adopted approach, in which task-specific datasets are used to further train pre-trained LLMs. Distillery [8] demonstrates that as LLMs advance, explicit schema linking becomes less critical; in their experiments, a supervised fine-tuned (SFT) GPT-4o achieved the best results. To improve NL2SQL reasoning in complex domains such as finance and healthcare, SQL-R1 [9] introduces a novel reasoning model, trained via reinforcement learning (RL). DAIL-SQL [10] leverages GPT-4 with effective prompt engineering, delivering strong performance on the Spider benchmark. CodeS [11] customizes an LLM for Text-to-SQL tasks by pre-training StarCoder [12] on a large Text-to-SQL corpus, achieving solid results on datasets like BIRD [2]. XiYan-SQL [13] proposes a unique schema representation called M-Schema and applies fine-tuning to both a base LLM and a SQL selection model. DTS-SQL [14] presents a two-stage, multi-task fine-tuning framework for Text-to-SQL, incorporating a schema-linking pre-generation task before final SQL generation. The ROUTE framework [15] integrates supervised fine-tuning across noise filtering, data synthesis, and continued writing tasks, demonstrating the effectiveness of its multi-task tuning pipeline through collaborative prompting.

Fine-tuning-based methods present several key limitations. First, they require substantial computational resources and specialized hardware, which can be a significant barrier to adoption. Second, achieving optimal performance demands carefully designed pipelines—for sequentially refining pre-training objectives, ensuring data diversity, and tailoring models to task-specific requirements. This complexity makes it challenging for open-source models to match the multifaceted capabilities of proprietary LLMs in Text-to-SQL tasks. Furthermore, as many advanced LLMs, such as the GPT series, have recently become closed-source, the progress of fine-tuning-based methods faces notable dilemmas: they can no longer leverage state-of-the-art models.
